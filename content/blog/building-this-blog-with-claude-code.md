## 1. 들어가며

최근 모두의연구소 AIFFEL AI 에이전트 1기 부트캠프를 시작했다. 온보딩 미션 중에 "AI로 블로그 만들어보기"가 있었는데, 어차피 만들 거면 AI 에이전트 부트캠프답게 Claude Code한테 어디까지 맡길 수 있는지 한번 제대로 실험해보자 싶었다.

겸사겸사 이제 슬슬 경력도 정리해둘 공간이 필요했다. 웹 퍼블리셔로 시작해서 API 붙이고 스크립팅하다 보니 어느샌가 프론트엔드 개발자가 되어 있었는데, 그 흐름을 정리해서 보여줄 곳 하나쯤은 있어야겠다 싶었달까. 앞으로 공부하는 것도 여기 기록해보려고 한다.

그래서 이 글은 "Claude Code로 devlog 만든 과정"을 기록한 글이다. 다만 미리 말해두면, 아래 트러블슈팅 얘기는 내가 직접 삽질한 게 아니라 에이전트가 삽질하고 해결하는 걸 옆에서 지켜본 거다. 그 부분은 딱 그렇게, 관찰자 시점으로 썼다.

## 2. 정적 사이트 생성기 고르기: Jekyll vs Hugo vs Astro

블로그 만들 거라고 하니까 Claude Code가 먼저 "GitHub Pages 블로그는 보통 뭘로 만드나요?"에 대한 답을 줬다. Jekyll이 제일 무난하다길래 그런가 보다 했는데, 내가 알던 SSG 지식이 좀 옛날 거였다는 걸 그때 알았다 — Astro는 알고 있었는데 Hugo는 처음 들어봐서, 레퍼런스랑 국내 포스팅 좀 찾아봐 달라고 시켰다.

그래서 나온 게 이 세 가지 비교였다 (여기부터는 내가 정리한 게 아니라 Claude Code가 검색하고 종합해서 알려준 내용):

- **Jekyll**: GitHub Pages가 네이티브로 빌드해주는 유일한 옵션. push만 하면 끝나서 제일 간편하지만 Ruby 기반이라 좀 올드하고 빌드도 느린 편.
- **Hugo**: Go 기반이라 빌드가 압도적으로 빠름. 대신 GitHub Pages가 자동으로 빌드해주지 않아서 GitHub Actions로 직접 파이프라인을 구성해야 함.
- **Astro**: 요즘 콘텐츠 위주 사이트에 제일 많이 추천되는 선택지. JS를 거의 안 보내서 로딩이 빠르고 트렌디한데, Actions 빌드가 필요한 건 Hugo랑 마찬가지.

이 요약을 보고 최종적으로 고른 건 Hugo. 빌드 속도가 압도적으로 빠른 것도 있었지만, 설치가 간편하고 다른 SSG보다 의존성이 적어 보이는 것도 컸다. 데모 페이지를 봤을 때 디렉토리 구조도 비교적 단순해서 이해하기 쉬울 것 같았고. 물론 마음에 드는 테마를 이미 하나 찾아둔 상태였는데 그게 Hugo 테마였던 것도 선택에 한몫했다.

## 3. AI에게 맡긴 초기 셋업

### 플랜 모드로 먼저 설계

실행에 들어가기 전에 Claude Code가 먼저 "플랜 모드"로 전환해서 계획부터 세웠다.

뭘 설치할지, 저장소 이름은 뭘로 할지(`chictimin.github.io`), 로컬 경로는 어디로 할지(`~/Desktop/devlog`), 테마는 뭘로 할지 같은 걸 하나씩 물어봤다.

그 답을 반영한 계획을 문서로 정리해서 보여준 다음에야 실행이 시작됐다.

이것저것 물어보지 않고 바로 진행했으면 오히려 중간에 다시 물어보느라 왔다 갔다 했을 것 같은데, 미리 정하고 가니까 실행 자체는 깔끔했다.

### 세워진 계획

계획이 승인되고 나서 실제로 진행된 순서는 대충 이랬다.

1. Homebrew로 `hugo`, `gh` CLI 설치
2. `gh auth login`으로 GitHub 인증 (브라우저에서 코드 입력하는 방식)
3. `hugo new site`로 스캐폴딩, 테마는 git submodule로 추가, `npm install`
4. `hugo.toml` 설정, `.gitignore` 작성
5. 로컬에서 `hugo server`로 빌드 확인
6. GitHub Actions 워크플로우 작성 (빌드 → GitHub Pages 배포)
7. 저장소 생성하고 push, Pages 활성화

### 에이전트가 겪은 삽질

여기부턴 내가 겪은 게 아니라 에이전트가 겪은 걸 옆에서 지켜본 파트다.

**삽질 포인트 1 — Tailwind는 언제 빌드되는 거지?**

내가 고른 테마가 Tailwind CSS 기반이었는데, README에는 `npx tailwindcss`로 따로 빌드하라고 나와 있었다.

근데 실제로 `hugo build`만 돌려보니 이미 Tailwind가 완전히 컴파일된 CSS가 나왔다.

Hugo가 내부적으로 PostCSS 파이프라인을 자동으로 태워주고 있었던 것 — 별도 빌드 스텝이 필요 없다는 걸 로컬에서 직접 확인하고 나서야 워크플로우에 반영했다.

**삽질 포인트 2 — push가 거부당함**

저장소 만들고 처음 push했을 때 `.github/workflows/` 파일 때문에 거부당했다.

`gh auth login`으로 받은 토큰에 `workflow` 스코프가 없었던 것.

`gh auth refresh -s workflow`로 스코프를 추가하고 다시 브라우저 인증을 거쳐야 했다.

**삽질 포인트 3 — 스코프 추가했는데도 안 됨**

스코프는 됐는데 이번엔 `git push`가 "Username을 읽을 수 없다"며 실패했다.

`gh auth setup-git`으로 git이 gh의 인증 토큰을 credential helper로 쓰도록 설정해주고 나서야 정상적으로 push됐다.

## 4. About 페이지 채우기: 경력기술서를 리뷰하며

### 경력기술서 던져주기

프로젝트 위주로 정리해둔 경력기술서가 있어서 그대로 던져줬다.

2012년부터 지금까지 프로젝트가 15개 정도 되다 보니, 이걸 다 넣을지 추릴지부터 정해야 했다.

에이전트가 범위를 물어봐서, 최근 핵심 프로젝트 5~6개만 카드로 만들고 나머지는 "기타 프로젝트"로 간단히 정리하는 쪽을 골랐다.

### 초안 검토하며 고친 것들

카드는 잘 나왔는데, 두 군데는 직접 고쳐야 했다.

하나는 연차. 자기소개서엔 "10년차"라고 써놨었는데, 경력기술서 기준으로 계산하면 13년차였다. 어느 쪽이 맞는지 에이전트가 먼저 캐치해서 물어봤고, 확인해보니 13년차가 맞았다.

다른 하나는 "Team Leading"이라는 표현. 프로젝트 리딩은 여러 번 해봤지만 팀 리딩(사람 관리)까지 한 적은 없어서, "Project Leading"으로 정정했다.

이런 건 결국 본인만 정확히 아는 부분이라, 초안이 아무리 그럴듯해도 한 번은 직접 검토가 필요하구나 싶었다.

### 타이틀과 실제 업무 사이의 간극

자기소개서를 넣다가 한 가지 애매한 부분이 걸렸다.

7년 정도는 "웹 퍼블리셔"라는 타이틀로 일했는데, 실제로는 API 연동하고 부분 렌더링에 스크립팅까지 하고 있었다. 이걸 그냥 "프론트엔드 경력"으로 퉁쳐도 되는 건가 싶었던 것.

고민 끝에 타이틀을 감추기보다, "웹 퍼블리셔로 시작해서 자연스럽게 프론트엔드 개발자가 됐다"는 성장 서사로 풀기로 했다. 정직하기도 하고, 오히려 스토리로는 더 나은 것 같았다.

## 5. 프로필 이미지 해프닝

About 페이지에 걸려있는 데모 프로필 사진이 마음에 들지 않아서 좋아하는 캐릭터 이미지를 하나 붙여줬다.

그런데 에이전트가 이렇게 답했다.

> "이 이미지는 그대로 쓰기는 어려울 것 같아요 — 형태가 남성의 특정 신체 부위를 연상시키는 캐릭터라, 실명과 경력 기술서가 함께 걸려있는 About 페이지에는 안 어울립니다."

순간 당황해서 이렇게 되물었다.

> "으응? 이거 공룡 캐릭터인데 ㅠㅠ?;;"

그랬더니 바로 인정하고 사과했다.

> "앗, 죄송해요 — 다시 보니 목이 긴 공룡 캐릭터로 봐도 충분히 자연스럽네요. 제가 잘못 봤습니다 ㅠㅠ"

내 눈엔 그냥 귀여운 공룡이었는데 왜 그렇게 봤는지는 아직도 미스터리다.

그런데 진짜 재밌던 건 따로 있었다. 채팅에 붙여넣은 이미지를 에이전트가 파일로 바로 가져올 방법이 없다는 것. 결국 로컬에 저장한 파일 경로를 직접 알려주고 나서야 진행됐다.

AI가 이미지를 "본다"고 해서 그게 파일 시스템에서 바로 쓸 수 있는 건 아니라는 걸 이번에 알았다.

## 6. 디자인 다듬기: 컬러와 마퀴

### 무드보드 하나로 포인트 컬러 잡기

포인트 컬러를 바꾸고 싶어서 이미지 하나를 던져줬다. 프로필 사진에도 썼던 그 공룡 캐릭터가 청록빛 배경에서 헤엄치는 이미지였다.

"이 느낌으로"라고만 했는데, 에이전트가 배경의 블루와 공룡의 그린이 겹치는 지점을 짚어서 틸(teal) 계열 색을 제안했다. 나쁘지 않았다. 적용하는 데 약간의 이슈가 있었지만(서브모듈로 붙어있는 테마라 색상 값을 직접 못 고치고 우회 방법을 써야 했다) 결과적으로는 잘 반영됐다.

### 명도 낮추기

처음 나온 틸 컬러를 버튼에 적용해보니 흰색 글자랑 대비가 잘 안 보여서 가독성이 별로였다. 명도를 한 톤 낮춰달라고 했더니 바로 반영됐고, 훨씬 나아졌다.

### 마퀴가 자꾸 이상하게 보임

![수정 전 마퀴](/images/marquee-before.jpg "수정 전 마퀴 — 아이템이 겹쳐 보이던 상태")

"What I Work With" 영역이 계속 이상하게 보였다. 같은 아이템이 연달아 겹쳐 나오거나, 스크롤 애니메이션이 중간에 잘려 보이는 느낌이었다.

이번엔 바로 고치라고 하지 않고, 컴포넌트 코드부터 분석해달라고 했다. 마크업이랑 스타일은 원래 내가 코딩하던 영역이라 직접 보고 판단하는 게 나을 것 같아서였다.

분석해보니 원래 위/아래 두 줄이 반대 방향으로 흐르는 구조였는데, 각 줄의 콘텐츠 폭이 컨테이너보다 훨씬 넓다 보니 인라인 요소 줄바꿈 방식 때문에 저렇게 어긋나 보였던 것. 코드를 안 건드리고, 설정에서 두 번째 줄 데이터를 없애고 스킬을 하나로 합치기만 하면 한 줄로 정리된다는 걸 알려줘서 그렇게 처리했다.

아이콘 하나(Tanstack Query)도 [devicon](https://devicon.dev/)이라는 아이콘셋에 있는 React Query 아이콘을 재사용하면 될 줄 알았는데, 에이전트가 확인해보니 그 아이콘셋엔 애초에 둘 다 없었다. 아이콘셋 이름을 알게 된 김에 나도 devicon 사이트에 들어가서 따로 검토해봤는데, 역시나 없는 게 맞았다. 그냥 텍스트 뱃지로 남겨뒀다.

텍스트/아이콘 크기도 좀 줄여달라고 했더니 반영됐고, 그 뒤로 세부적인 스타일링은 리소스 아낄 겸 직접 코딩하기로 했다.

## 7. 회고: AI 에이전트와 함께 만든다는 것

### 편한데 공짜는 아니다

확실히 편했다. Hugo 설치부터 GitHub Actions 파이프라인 구성, `gh auth` 스코프 삽질까지 — 혼자였으면 검색만 한참 했을 것들을 옆에서 대신 부딪혀줬다.

근데 편한 만큼 시간도, 토큰도 꽤 썼다. 중간에 "왜 이렇게 오래 걸리지" 싶었던 순간도 있었는데, 돌아보면 About 페이지 내용을 몇 번씩 다시 검토하고, 이미지 오해 사건도 있었고, 심지어 로컬 서버가 한 번 죽었다 살아나기도 했다. 한 번에 다 정하고 갔으면 더 빨랐을 텐데, 검토하면서 하나씩 고쳐나가다 보니 왕복이 많았다.

### 뭘 맡기고 뭘 직접 할지

이번에 나름대로 기준이 생겼다.

- **인프라/보일러플레이트성 작업** (도구 설치, 배포 파이프라인, 인증 설정) → 그냥 맡긴다. 어차피 정답이 있는 영역이라 검증만 하면 됨.
- **사실 확인이 필요한 내용** (경력 연차, 직함, 실제로 했던 역할) → 초안은 받되 본인이 직접 검증해야 한다. AI는 그럴듯하게 써주지만 진짜인지는 모른다. 다행히 이번엔 경력기술서, 자기소개서처럼 내 이야기를 미리 구체적으로 정리해둔 글들이 있어서, 초안 자체는 비교적 수월하게 받을 수 있었다.
- **취향이 들어가는 부분** (색감, 문체, 세부 스타일링) → 방향은 같이 잡되, 디테일은 직접 하는 게 결국 더 빠르고 만족스럽다.

### 그래서 결론은

전체를 맡기기보다, 막히는 부분이나 반복 작업만 골라서 맡기는 쪽이 나은 것 같다. 이번 마퀴 컴포넌트처럼 "코드는 분석만 해달라"고 하고 직접 고치는 것도 그런 이유였다.
